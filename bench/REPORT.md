# Plugin Verifier performance investigation — report

Investigation performed 2026-07-07 on the branch base `35b2c79` with the
harness in this directory. Environment: Linux container, OpenJDK 21.0.10,
4 verification worker threads, Gradle 8.14.3 (`GRADLE_BIN`).

## Benchmark setup

Corpus (deterministic, generated by `bench/gen.py --out bench/work/corpus`,
seed 42): a synthetic IDE (`build.txt` IU-231.9999 + `lib/idea-core.jar`
with 1600 API classes and an IDEA CORE descriptor) and 6 plugin ZIPs with
250/500/750/1000/1400/2100 classes (6000 total) that heavily reference the
IDE API, including deprecated members and references to classes absent from
the packaged IDE (so problem/usage registration paths are exercised: 301
deprecated-API usages, and 240 classes referencing a missing package, which
the verifier aggregates into one compatibility problem per plugin).

Command under test (one JVM, all 6 plugins):

```
java -jar verifier-cli-dev-all.jar check-plugin @plugins.txt <ide> -offline \
  -verification-reports-dir <out> -verification-reports-formats plain,html,markdown
```

## Baseline (before change)

Medians of 5 runs (`bench/bench.py run --label baseline`):

| Metric | Baseline |
|---|---|
| wall clock | 6.00 s |
| CPU user + sys | 17.39 s + 1.89 s |
| peak RSS | 818 MB |
| verifier-reported verification time | 4674 ms |

JFR phase attribution (execution samples at 1 ms period, all threads):

| Phase | Share |
|---|---|
| bytecode/class scanning | **61.1 %** |
| classpath construction | 23.0 % |
| plugin archive parsing | 7.3 % |
| verifier-other | 4.6 % |
| JVM/other | 2.6 % |
| report generation | 1.2 % |
| dependency resolution | 0.2 % |

## Bottleneck identification

Bytecode scanning dominates. Within it, the hottest project frames were
`MethodAsm.getParameterNames` / `MethodAsm.getLocation` — **5.6 % of all
CPU samples** were inside `MethodAsm.location` construction
(`bench/bench.py hot ... --phase bytecode-scan`). Stack analysis showed the
cost is *method parameter-name extraction* (sorting the local variable
table + parsing the method descriptor + list allocation), triggered from
places that do not need parameter names at all:

1. `ClassLocationApiUsageFilter` (runs per scanned instruction for every
   API usage candidate) built full `MethodLocation`s for the caller and the
   invoked method, then immediately discarded everything except
   `containingClass`.
2. `AbstractMethodVerifier` (runs for every verified class) walked the
   whole parent hierarchy and built a `MethodLocation` for **every method of
   every parent class**, though the values are only needed for the rare
   `MethodNotImplementedProblem`; for `implementedMethods` the values were
   *never read* (only the key set is used).
3. `MethodAsm.methodParameters` has a memoization field
   (`_methodParameters`), but three of its four branches returned before the
   cache was assigned, so the intended per-instance caching only engaged on
   a rare fallback path.

Two other hotspots were measured and deliberately left alone (see
"Remaining unoptimized" below): `CacheResolver.resolveClass` lookup
overhead (~10 % of samples) and one-time `JdkJImageResolver` initialization
(~8 % of samples on this short run).

## The optimization (one logical change)

*Do not materialize member locations (with parameter-name extraction) on
hot verification paths that don't need them.* Three small cuts, ~25 changed
lines, no public API changes:

- `MethodAsm.kt`: make all branches of `methodParameters` flow into the
  existing memoization field (remove three early `return`s).
- `ClassLocationApiUsageFilter.kt`: obtain the containing class location as
  `containingClassFile.location` instead of `location.containingClass` —
  identical value by construction (`MethodLocation.containingClass` *is*
  the `hostClass` passed from `containingClassFile.location`), without
  building the member location. The existing argument order of the field
  overload (usage/api naming) is preserved exactly.
- `AbstractMethodVerifier.kt`: store `Method` objects and a signature set
  instead of eager `MethodLocation` maps; materialize a location only for
  reported unimplemented methods. Map-overwrite semantics ("last visited
  parent wins") and iteration order are preserved.

## Post-change numbers

Medians of 5 runs (`bench/bench.py run --label optimized`) and stable JFR
metrics:

| Metric | Baseline | Optimized | Delta |
|---|---|---|---|
| CPU samples inside `MethodAsm.location` construction | 5.63 % | 1.98 % | **−65 %** |
| total allocated bytes per run (`jdk.ThreadAllocationStatistics`, 3 runs each) | 2.30–2.35 GB | 2.13–2.20 GB | **≈ −7 %** |
| CPU user (median) | 17.39–18.25 s | 17.65–17.82 s | ≈ −2 % (within noise) |
| wall clock (median) | 6.00–6.22 s | 6.05–6.11 s | within noise |
| peak RSS | 818–947 MB | 812–947 MB | unchanged |
| verifier-reported verification time | 4674–4896 ms | 4690–4783 ms | ≈ −1..−2 % (within noise) |

An interleaved A/B experiment (10 alternating pairs) put the paired
wall-clock delta at −0.85 % median with ±30 % spread — this container
cannot resolve a ~2 % end-to-end change, which is why the sample-attribution
and allocation figures above are the primary evidence. The remaining
`getMethodParameters` samples (~2.5 %) come from call sites that genuinely
need parameter names (deprecated/experimental usage objects that appear in
reports).

## Why the change is safe

- The values produced are provably identical: the filter compares the same
  `ClassLocation`s (equality is `className`-based and the instances are
  constructed from the same class file), and `AbstractMethodVerifier`
  derives the same `MethodLocation` from the same stored method, just
  later and only when needed.
- Memoizing `methodParameters` matches the code's existing intent (the
  cache field and fallback-branch assignment already existed); inputs
  (`MethodNode`) are not mutated during verification.
- **Parity gate**: `bench/bench.py parity` re-runs both jars over the whole
  corpus — all 32 verification report files (plain, HTML, markdown,
  telemetry) are byte-identical after normalizing run timings only. The
  corpus produces 301 deprecated-API usages and 6 missing-package
  compatibility problems, so problem texts and usage locations (including
  parameter names in report presentation) are covered.
- Unit tests: `MethodAsmParametersTest` (new, in `verifier-test`) covers
  the regular / no-LVT / enum-constructor parameter branches, annotation
  offsets, memoization, and `location.parameterNames`.
- Full `verifier-test` suite passes (`:intellij-plugin-verifier:verifier-test:test`).

## Remaining unoptimized (documented, out of scope)

- `CacheResolver.resolveClass` lookup overhead (~10 % of samples): Caffeine
  cache + string-key hashing per class resolution; improving it means
  changing the resolver design.
- `JdkJImageResolver` construction (~8 % on this corpus): one-time JDK
  jimage indexing per run; amortized on large corpora, and making it lazy
  would change when JDK problems surface.
- `MethodAsm`/`FieldAsm` instances are re-created per resolution, so the
  parameter cache does not amortize across separate resolutions of the same
  method (fresh instances), only within one.
- Legitimate location construction for reported usages (deprecated,
  experimental, internal) still extracts parameter names — those appear in
  report output and must stay.

## Reproduce

```bash
./gradlew :intellij-plugin-verifier:verifier-cli:shadowJar
python3 bench/gen.py --out bench/work/corpus
python3 bench/bench.py run --label <label> --runs 5
python3 bench/bench.py compare bench/work/results-baseline.json bench/work/results-optimized.json
python3 bench/bench.py parity --jar-a <old-all.jar> --jar-b <new-all.jar>
```

CI: `.github/workflows/verifier-bench.yml` (manual dispatch) runs the same
steps and uploads results + parity verdict. A hard time-based regression
gate is intentionally not enforced (noisy shared runners); this report and
the committed commands serve as the documented baseline.
