# Plugin Verifier performance benchmark harness

A reproducible benchmark for `verifier-cli check-plugin` over a synthetic,
deterministic corpus of plugin archives. It measures wall-clock time, CPU
time and peak RSS, and captures JFR profiles whose execution samples are
attributed to verifier phases:

- **plugin-parsing** (`plugin.structure.intellij/base/ide`, JDOM, Xerces)
- **classpath-construction** (`plugin.structure.classes/jar`, resolvers, JDK image)
- **bytecode-scan** (ASM, `pluginverifier.verifiers`, usage processors/filters)
- **report-generation** (`pluginverifier.output`, reportage)
- plus `dependency-resolution`, `verifier-other`, `jvm-other`

No binaries live in git: the corpus is generated by `gen.py` (seeded, fixed
archive timestamps, deterministic for a given JDK).

## Requirements

JDK 11+ (`java`, `javac`, `jfr` on `PATH`; JDK 17+ recommended for
high-frequency JFR sampling), Python 3.8+, and a Gradle able to build the
repo (the wrapper, or `GRADLE_BIN=/path/to/gradle` where the wrapper cannot
download its distribution).

## Commands

```bash
# 1. Build the verifier under test
./gradlew :intellij-plugin-verifier:verifier-cli:shadowJar

# 2. Generate the corpus (deterministic; ~1600 IDE classes + 6 plugins / 6000 classes)
python3 bench/gen.py --out bench/work/corpus

# 3. Measure (5 timing runs + 1 JFR-profiled run)
python3 bench/bench.py run --label baseline --runs 5

# ... apply a change, rebuild the shadowJar ...
python3 bench/bench.py run --label optimized --runs 5

# 4. Compare as a markdown table
python3 bench/bench.py compare bench/work/results-baseline.json bench/work/results-optimized.json

# 5. Behavior-parity gate: normalized verification reports must be byte-identical
python3 bench/bench.py parity --jar-a old-verifier-all.jar --jar-b new-verifier-all.jar

# 6. Explore a profile
python3 bench/bench.py hot bench/work/run-baseline/profile.jfr --phase bytecode-scan
```

Only run-timing lines in `telemetry.txt` (and timestamps/durations in logs)
are normalized by `parity`; verdicts, problems, warnings, plugin sizes and
verified-class counts are compared byte-for-byte.

In CI, the same commands run via `.github/workflows/verifier-bench.yml`
(manual dispatch), which uploads `results-*.json` and the parity verdict as
artifacts.

## Measurement notes

- Wall/CPU/RSS come from `resource.getrusage(RUSAGE_CHILDREN)` around a
  dedicated JVM per run; medians over `--runs` are reported.
- Wall-clock deltas below ~5% are not resolvable on noisy shared runners;
  use the JFR phase attribution and the total-allocation figure
  (`jdk.ThreadAllocationStatistics`) as the stable signals, and treat the
  documented baseline in [REPORT.md](REPORT.md) as the reference numbers.
  A hard-failing time-based regression test is deliberately not provided —
  it would be flaky on shared hardware.
